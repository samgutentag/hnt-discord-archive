# HIP 66: Trust Score & Denylist Convenience - Author(s): @HBaguette - Start Date: 2022-06-10 - Category: Technical - Original HIP PR: https://github.com/helium/HIP/pull/430 - Tracking Issue: https://github.com/helium/HIP/issues/438 - Status: In Discussion # Summary There is no single way to determine whether a hotspot is spoofing. This HIP therefore looks at several types of data that are typically looked at when looking for spoofers and combines them into a score, the Trust Score, which is designed to tell how likely a hotspot is to be spoofing, in order for them to be manually added to the denylist later. The HIP doesn't make any automatic additions to the denylist, nor does it invalidate witnesses that it deems to be spoofing ; it only attributes a Trust Score to each hotspot. Additionally, the HIP provides convenient ways for hotspot owners to report hotspots to the denylist as well as prove their own integrity, directly in-app. # Rendered View https://github.com/helium/HIP/blob/main/0066-trust-score-and-denylist-convenience.md

From the HIP
* The Helium app should allow users to prove their GPS localization directly from the Helium app.
This is trivial to spoof in many ways: Off the top of my head: Download AndroidStudio for free, run the Helium app in the Android simulator, fake the GPS using the simulator controls. There are other ways that are much harder to detect.
*The Helium app should be able to tell whether there are other apps running in the background, and whether a device is rooted.
This is first thing is not possible to do reliably. The 2nd one is possible but it's a bit of an arms race, new ways to avoid rooted detection, then libraries that detect are updated, then the cycle repeats. Trusting data generated from the phone is a deep and tricky problem. A common approach in applications that need to secure data involving a smart phone is to either pass encrypted data from one remote server to another through the phone in a way that is tamper evident (signed messages) or to do some sort of proof of work to give you more confidence the phone really did do the right steps to calculate whatever data it give you. 0weavern 07/08/2022 5:21 AM
I like this idea. I've implemented similar 'risk' scores using analytics data at previous businesses. One thing I would add is a weighting on each datapoint. The HIP seems to give individual datapoints a 0 or -1 on the trust score. When implementing these in the past, I've tended to use a sliding scales from 1 - 100. That way a small, one off infraction (ie a witness to far) would give that data point a 99 score, but if it happens multiple times, it would be lower number like 80, increasing it's risk. That should minimise honest hotspot owners being caught up if it's another hotspot causing the issue, or a one off IP address change, or further hotspot deployments that aren't in a cluster.
05:23
Additionally - some datapoints are more indicative of spoofing than others. For instance - a VPN with an IP address no in the asserted location could be fairly low risk, but those who only witness each others hotspots would be high. Calculation could look something like: (datapoint_1_score * 1) + (datapoint_2_score * 0.5) + (datapoint_3_score * 1.5) = trust_score

HBaguette 07/08/2022 5:55 AM
Instead of weighting each datapoint thisway, I instead gave them different time periods, depending on the severity of the infraction (chosen subjectively for now). For example, high RSSI values are checked for the last 60 days whereas IP addresses, which are more likely to be false positives, are only checked for the last 7 days. Additional weights like you suggest can definitely be implemented. The current calculations are more an example of what can be done than the actual finit calculation. The priority should be to determine what data we can work with, and when we have a finite list, we can decide how they will all be taken into consideration for the calculation of the Trust Score.

So, FWIW, I'm developing a gadget that would ideally use LoraWAN Class C (or possibly B) to allow downlinks without immediately preceding uplinks. I don't particularly need or want IP, since I'm routing directly with MQQT. My current plan to use Helium, with its class A limitations, is to have the gadget uplink every 10 seconds to poll for incoming data. This requires 8,640 uplinks per day at a cost of $0.09 (assuming the poll requires 24 bytes or less), or $2.59 per month. That suits my current business case okay, but it's not ideal for two reasons:
1
09:55
1. My device uses more power uplinking every 10 seconds and staying awake for the several-second downlink response window, giving my device a duty cycle of about 30%. If I could do class B, I could have the same latency at a fraction of the duty cycle and power consumption. If I could do class C, I could be in receive-only mode much of the time. I'm not sure what this would do to power, although class C is supposed to be for devices with plenty of power. 2. Something around $0.10 per day seems fair, and I'm happy to pay the miners that much when my device roams near them. But it doesn't seem like my polling technique is the most efficient use of air time for them or for me. I'd just as soon pay them the same amount for the "possibility" of receiving a downlink, even if it doesn't ever occur.

krispyrice9 12/18/2021 9:19 AM
The forwarding idea is essentially what I had in mind, sort of an IP to to LoRaWAN bridge where only the payload is forwarded. The HTTP Custom Integration in the Helium console is similar to this idea with several limitations that solve the packet overhead you described. The integration endpoint URL forces a single layer 3 destination (endpoint host), TCP as the layer 4, and merges layers 5 through 7 into an HTTP REST endpoint. An additional limitation is that incoming messages (from the endpoint to the device) are scheduled rather than immediately sent - presumably until the next time that the class-A device uplinks? (I haven't played with this yet). I can imagine 2 intermediate steps to something like IP-over-Helium could be: 1) Create a new integration type of "multi-HTTP", which allows the developer to enumerate multiple endpoints that the uplinking device selects (consumes a few bits), and permits incoming downlinks from multiple endpoints, possibly the same list. 2) Advocate for LoRaWAN Class B (maybe even C) so that downlinks can be sent deterministically. This wouldn't provide true layer-3 IP solution. But it would provide a more flexible integration solution.

wolfenhawke 12/22/2021 1:32 PM
I think IETF is looking to have a consolidating server (called the Application Server) that terminates the end device Lora traffic, packetizes, and then onramps onto the IP network. While this will work for this HIP, my understanding originally was this intended to more easily send data packets that could be directly consumed by an IP system. This didn't seem like something realistic for full IP traffic or HTTP. BUT, to embed something like an MQTT structure may be possible. MQTT is made for small payload systems anyway. So, if we had a software (firmware?) option that allows routing MQTT encapsulated traffic, that would cover this. Further, it would seem this may be achievable with only the actual end sensor doing this and not the gateway. If this was done economically enough (packet size), then the helium gateways wouldn't know the content, wouldn't care, and would just pass it along. The end application of the OEM would unencapsulate, take the MQTT payload and then do with it what they wanted. This would not require Semtech involvement, and may meet the spirit of the requirement. I don't know the encapsulation packet size and format and if it could be made to work. Something to look into.

Ioannis | Kudzu 08/01/2021 1:03 AM
Hello guys! The proposal here is not by any means connected to any commercial service. The HIP merely tries to ensure that whatever data comes out of this meta-data stream is safe for consumption and approved by all the Helium stakeholders. Note that, as we mention on the HIP, anyone that owns a helium hotspot can already tap into the analytics stream by inserting a man-in-the-middle proxy, right before the gateway client. (We even have customers doing that with DIY miners, because they wanted to debug sensor coverage problems in some areas), and this is raising some legit security concerns. Therefore, even if this does not change the actual blockchain code, it does introduce a new technology in the helium component stack. The benefit of this being a HIP is that: * It is an open discussion for all Helium stakeholders to express their opinion * Once settled will help helium further increase the security in the component stack (eg. disallowing such tampering with the stream) * At the same time it will enable large-scale deployment operators to re-use existing monitoring technologies (remember that this is already happening in the LoRaWAN world) * Once it becomes a "standard" technology, the same meta-data can be extracted from other wireless protocols (edited)

Ioannis | Kudzu 08/05/2021 6:54 AM
to kick-off a bit the discussion and drive things further, I am reminding the remaining open issues from the HIP document:
(1) We need to decide weather we go with JSON (and therefore creating a backwards-compatible interface, similar to the semtech UDP packet itself), or we go with Protobuf, and therefore breaking any existing solution.
(2) Some discussion might be needed to further clean-up the fields in the analytics protocol. More specifically, if there is any smart alternative to encode the different fields for LORA or FSK encoding.
Ioannis | Kudzu 08/05/2021 7:02 AM
Regarding (1) I am all forward to maximise compatibility with the existing UDP streams from Semtech but I am afraid that this leans towards unnecessary back-and-forth conversions. eg: 1. The data arrive in this packet_forwarder format 2. They MUST be parsed in order to be processable, transforming them in an in-memory representation 3. High-level conversions functions (such as parsing string timestamps, etc.) has already occurred, and the data are kept in the highest possible detail. Converting them back into packet_forwarder format sounds like a step backwards. Meaning that we would have to encode timestamps back in string formats, expand information that are already compacted and organised in memory etc. Therefore I see little value into doing this conversion and I would suggest to jump directly towards the closest thing in memory representation, that being Protobuf.
07:04
If we still want to maintain a packet_forwarder compatibility layer, it's trivial to write an application that runs locally, next to the gateway client component that: 1. Subscribes to the local analytics stream 2. For every packet received, it performs the transformation routines needed 3. Forwards it in an 100% compatible packet_forwarder UDP frames to the desired target
